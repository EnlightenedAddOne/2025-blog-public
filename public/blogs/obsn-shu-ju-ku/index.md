### 第二章 关系模型及其操作

#### 笛卡儿积与关系（P44）

- **定义：**
  给定 ​*​n 个不同的集合*​​ `D₁, D₂, ..., Dₙ`（这些集合称为​​*域*​​，在数据库上下文中，每个集合代表一个属性的所有可能取值），它们的笛卡尔积定义为：
  ​​`D₁ × D₂ × ... × Dₙ = { (d₁, d₂, ..., dₙ) | d₁ ∈ D₁, d₂ ∈ D₂, ..., dₙ ∈ Dₙ }`。

  假设集合A={a, b}，集合B={0, 1, 2}，则两个集合的笛卡尔积为{(a, 0), (a, 1), (a, 2), (b, 0), (b, 1), (b, 2)}。

- **理解：**
  1.  笛卡尔积表示多个域间元素的所有组合情况，代表了客观实际中域与域之间的联系。
  2.  笛卡尔积的结果仍是集合，集合中的元素为元组，元组中的每个分量（$d1$,$d2$,$d3$...$dn$），都来自笛卡尔积的计算对应的位置的域$Di$。
  3.  笛卡尔积包含的元组总数（即基数）等于所有参与集合大小的​​*乘积*​​。`|D₁ × D₂ × ... × Dₙ| = |D₁| × |D₂| × ... × |Dₙ|`
  4.  笛卡尔积的*元组间是无序的*，但元组内的分量位置是根据参与的笛卡尔积运算的域确定的，即*元组中的分量位置是有序的*。
  5.  可根据元组中出现的各个分量，推理得出参与笛卡尔积运算的域。同时，可将笛卡尔积的集合形式表示为二维表的形式。
      ​

#### 关系模式（P47）

- **定义：**
  在数据库中，若关系头由属性名称 $A_1, A_2, \ldots, A_n$ 组成，属性对应域为$D_1, D_2, \ldots, D_n$，则集合$\{A_1, A_2, \ldots, A_n: D_1, D_2, \ldots, D_n\}$称为关系的模式。在各属性类型已知或延后考虑时，也可将关系模式（Relation Schema）表示为属性集合形式$\{A_1, A_2, \ldots, A_n\}$。

#### 实体完整性及相关的码（P50）

- **实体完整性**
  - 定义： 关系模型中的一个​​基本规则​​。它要求：
    1. ​*​唯一性（Uniqueness）:​*​ 任何两个不同的元组（行），其主码的值都不能相同。
    2. ​*​非空性（Non-nullability）:​*​ 主码的任何属性（列）的值都不能是未知的（即不能是 `NULL`）。

  - 通俗理解：​​数据库表中的每一行都必须有一个“独一无二的身份证”，这个“身份证”上的每个信息栏都必须填满，不能为空。

- **各种“码（Key）”​**
  - ​**​超码​**​：任何能唯一标识元组的属性集（可能含冗余）。
  - ​**​候选码​**​：最小的超码（无冗余属性）。
  - ​**​主码​**​：被设计者选定为主标识的候选码（一个表只有一个）。
  - ​**​外码​**​：存在于本表中、​**​引用​**​另一个表（或本表）​**​主码或候选码​**​的属性集（用于建立表间关联）。

#### 常用 SQL 语句

##### 1. 增加数据 (Create)

使用 `INSERT INTO` 语句向表中添加新的记录：

```sql
--向一个名为 employees 的表中插入一条记录
INSERT INTO employees (first_name, last_name, age, department)
VALUES ('John', 'Doe', 28, 'Sales');
```

##### **2. 查询数据 (Read)**

- _使用 `SELECT` 语句来查询表中的数据：_

```sql
--查询 employees 表中所有部门为 'Sales' 的员工的名字和年龄
SELECT first_name, last_name, age
FROM employees
WHERE department = 'Sales';
```

- _使用`WHERE` 子句 - 过滤数据_

  ​​运算符：`<`,`>`,=, `/`,`!=`, `>`, `<`, `>=`, `<=`, `BETWEEN`, `LIKE` (通配符 `%` 任意字符, `_` 单个字符), `IN` (值列表), `IS NULL`, `IS NOT NULL`

```sql
SELECT * FROM Customers WHERE Country = 'Germany';
SELECT ProductName, Price FROM Products WHERE Price BETWEEN 10 AND 20;
SELECT * FROM Customers WHERE Country IN ('Germany', 'France', 'UK');
SELECT * FROM Customers WHERE ContactName LIKE '%Mar%'; -- 名字中包含"Mar"
SELECT * FROM Orders WHERE ShippedDate IS NULL; -- 未发货的订单
```

- _`ORDER BY` 子句 - 排序_

```sql
SELECT ...
FROM ...
[WHERE ...]
ORDER BY column1 [ASC|DESC], column2 [ASC|DESC], ...;
-- ASC 升序(默认), DESC 降序
```

##### 3. 更新数据 (Update)

使用 `UPDATE` 语句修改表中已有的记录：

```sql
--更新 employees 表中名字为 'John' 的员工所在的部门为 'Marketing'：

UPDATE employees
SET department = 'Marketing'
WHERE first_name = 'John'; --如果不使用 `WHERE` 子句，则表中所有记录都将被更新。
```

##### 4. 删除数据 (Delete)

使用 `DELETE` 语句从表中删除记录：

```SQL
--删除 `employees` 表中所有年龄大于60岁的员工：
DELETE FROM employees
WHERE age > 60;
```

##### 5. 创建对象(CREATE)

- _​创建数据库：_

```sql
CREATE DATABASE database_name;
```

​

- ​*创建表：​​（核心DDL语句）*

```sql
CREATE TABLE table_name (
    column1 datatype [constraint], -- 列名 数据类型 [约束]
    column2 datatype [constraint],
    ...,
    [CONSTRAINT constraint_name] CONSTRAINT_TYPE (column1, ...), -- 表级约束
);
```

##### 6. 删除对象(DROP)

- _删除表_

```sql
DROP TABLE table_name;
```

​

- ​*删除数据库*：​\*\*​

```sql
DROP DATABASE database_name;
```

### 第四章 关系型数据库理论

#### 属性闭包及求解算法（P133）

**人话：** 闭包就是由一个属性直接或间接推导出的所有属性的集合。

**例如：** f={a->b，b->c，a->d，e->f}；由a可直接得到b和d，间接得到c，则a的闭包就是{a，b，c，d}

**例题：** 已知关系模式 $R\langle U,F\rangle$，其中 $U=\{A, B, C, D, E\}$，$F= \{ AB\rightarrow C,\ B\rightarrow D,\ C\rightarrow E,\ EC\rightarrow B,\ AC\rightarrow B \}$，求 $(AB)_F^+$ 和 $(CE)_F^+$。

**解：**
**1. 求 $(AB)_F^+$**
令 $X = \{AB\}$

1. 扫描 $AB\rightarrow C$，$AB \subseteq X$，$X = \{A,B,C\}$
2. 扫描 $B\rightarrow D$，$B \subseteq X$，$X = \{A,B,C,D\}$
3. 扫描 $C\rightarrow E$，$C \subseteq X$，$X = \{A,B,C,D,E\}$
4. $X$ 包含全部属性，终止
   **结果：**
   $$(AB)_F^+ = \{A,B,C,D,E\}$$

**2. 求 $(CE)_F^+$**
令 $X = \{C,E\}$

1. 扫描 $AB\rightarrow C$：$AB \not\subseteq X$，$X$ 不变
2. 扫描 $B\rightarrow D$：$B \not\subseteq X$，$X$ 不变
3. 扫描 $C\rightarrow E$：$C \subseteq X$（$E$ 已存在），$X$ 不变
4. 扫描 $EC\rightarrow B$：$EC \subseteq X$，$X = \{B,C,E\}$
5. 扫描 $AC\rightarrow B$：$AC \not\subseteq X$，$X$ 不变
   **结果：**
   $$(CE)_F^+ = \{B,C,E\}$$

#### 候选码及其求解算法（P136）

**候选码定义：** 候选码是能够唯一标识关系中某一个元组的一个属性或属性集，且他的任意真子集不能唯一的标识一个元组。
例如：在学生关系中，学号可以唯一标识学生，班级+姓名也可以唯一标识一个学生，所有他们都是候选码。

**函数依赖集F的属性分类：**
L 类：仅出现在函数依赖左部的属性。
R 类：仅出现在函数依赖右部的属性。
N 类：在函数依赖左右两边均未出现的属性。
LR类：在函数依赖左右两边均出现的属性。

**例题：** 设关系模式R(A, B, C, D), 其函数依赖集：
F={D→B, B→D, AD→B, AC→D}，求R的所有候选码。
**解:**
L类: A, C
R类:
N类:
LR类: B, D
因为$(AC)_F^+ = \{A,B,C,D\}$，所以AC是R的唯一候选码

#### 第一、二、三范式（P146）

##### 第一范式（1NF）

**定义：​**​ 一个关系模式 `R` 属于 1NF，当且仅当 ​​*表中的每一列都是原子的*​​，即列值都是​​*不可再分*​​的基本数据项。1NF 禁止​​*多值属性*​​（一列中包含多个值）和​​*组合属性*​​（作为一列存储，但其值可分解为更基本的含义）以非原子形式存储。

- **违反 1NF 的例子：​**

| 订单号 | 客户 | 产品清单（违反！） |
| ------ | ---- | ------------------ |
| 1001   | 张三 | 产品A: 2; 产品B: 1 |
| 1002   | 李四 | 产品C: 3           |

- **规范化：**

| 订单号 | 客户 | 产品清单 |
| ------ | ---- | -------- |
| 1001   | 张三 | 产品A: 2 |
| 1001   | 张三 | 产品B: 1 |
| 1002   | 李四 | 产品C: 3 |

##### 第二范式（2NF）

**定义：​**​ 一个满足 1NF 的关系模式 `R` 属于 2NF，当且仅当 ​*​所有非主属性都完全函数依赖于 R 的主码​*​。（完全函数依赖P135）

**举例：** 比赛表 player_game ​，包含属性​：球员编号、姓名、年龄、比赛编号、比赛时间、比赛场地、得分。可知：候选键 = 主键 = (球员编号, 比赛编号)
​**​函数依赖关系​**​：1.主键决定的依赖​​：
(球员编号, 比赛编号) → (姓名, 年龄, 比赛时间, 比赛场地, 得分)
​​2.违反第二范式的部分依赖​​：
(球员编号) → (姓名, 年龄)
(比赛编号) → (比赛时间, 比赛场地)
**不满足第二范式​原因是：​** 1.非主属性（姓名, 年龄）部分依赖于候选键的子集（球员编号）2.非主属性（比赛时间, 比赛场地）部分依赖于候选键的子集（比赛编号）
**范式优化：**
需拆分为三个表以满足第二范式：1. ​​球员表​​ (球员编号, 姓名, 年龄) 2. ​​比赛表​​ (比赛编号, 比赛时间, 比赛场地) 3. ​​参赛记录表​​ (球员编号, 比赛编号, 得分)

##### 第三范式（3NF）

**定义：​**​ 一个满足 2NF 的关系模式 `R` 属于 3NF，当且仅当 ​​*不存在非主属性对候选码的传递函数依赖*​​。(即，不能存存非主属性A依赖于非主属性B，非主属性B依赖于主键C的情况，即存在“A→B→C”的决定关系）通俗地讲，该规则的意思是所有`非主键属性`之间不能有依赖关系，必须`相互独立`。

**举例：**

| 字段名称      | 字段类型      | 是否是主键 | 说明                 |
| ------------- | ------------- | ---------- | -------------------- |
| id            | INT           | ​**​是​**​ | 商品主键id（主键）   |
| category_id   | INT           | 否         | 商品类别id           |
| category_name | VARCHAR(30)   | 否         | ​**​商品类别名称​**​ |
| goods_name    | VARCHAR(30)   | 否         | 商品名称             |
| price         | DECIMAL(10,2) | 否         | 商品价格             |

    **不满足第三范式​原因是：​**
    	`category_name`（商品类别名称）依赖于 `category_id`（商品类别编号）​​，而非直接依赖于主键 `id`，非主键属性之间存在依赖关系。

### 第五章 数据库优化和管理

#### 事物的特征（P197）

1.  **原子性（Atomicity）​**
    - ​​定义​​：事务中的所有操作要么 ​​*全部成功执行*​​，要么 ​​*全部回滚撤销*​​（如同从未执行过）。
    - ​​作用​​：保证操作不可拆分，避免部分操作失败导致数据不一致。
    - ​​示例​​：
      转账事务包含 ​​*A账户扣款*​​ 和 ​​*B账户收款*​​ 两个操作。
      ✅ 成功：两个操作同时生效。
      ❌ 失败：若B账户收款失败，则A扣款操作自动撤销。
2.  **​ 一致性（Consistency）**
    - ​​定义​​：事务执行后，数据库必须从一个 ​​*合法状态*​​ 迁移到另一个 ​​*合法状态*​​，满足预定义约束（如主键唯一、外键关联、余额非负等）。
    - ​​作用​​：防止数据逻辑错误。
    - 示例​​：转账后总资金必须守恒：A减去的金额必须等于B增加的金额。
3.  **隔离性（Isolation）**
    - ​​定义​​：并发执行的事务之间 ​​*互不干扰*​​，每个事务感知到的数据状态是独立的（如同串行执行）。
4.  **持久性（Durability）​**​
    - ​**​定义​**​：事务一旦提交，其结果必须 ​**​永久保存​**​，即使系统故障（如断电、崩溃）也不丢失。
    - ​**​作用​**​：确保数据永久可靠。

#### 并发操作与数据的不一致性（P198）

1. **丢失更新（Lost Update）**
   ​场景​​：两个事务同时读取并修改同一数据，后提交的事务覆盖了前者的修改。
2. **污读（Dirty Read）**
   ​​场景​​：事务A读取了事务B ​​*未提交的数据*​​，随后事务B回滚，导致事务A读取到无效数据。
3. **不可重复读（Non-Repeatable Read）**
   ​场景​​：事务A多次读取同一数据，其间事务B修改了该数据，导致事务A前后读取结果不一致。

#### 封锁和封锁协议（P200）

##### 封锁的类型​

1. **排他锁（Exclusive Lock, X锁）**
   ​​用途​​：用于修改数据
2. **共享锁（Shared Lock, S锁）**
   ​​用途​​：用于读取数据

##### 封锁协议

1. **一级封锁协议（解决丢失更新）​​**
   ​​*规则*​​：​​写数据前必须加X锁​​，事务结束后释放（无论提交或回滚）。 读数据不强制加锁。
   ​​*作用*​​：避免​​丢失更新​​（两个事务同时修改同一数据，后者覆盖前者）。
   ​​*缺陷*​​：不解决脏读、不可重复读问题。
2. **二级封锁协议（解决污读）​​**
   ​​*规则*​​：在一级基础上，​​读数据前必须加S锁​​，读完后立即释放S锁（不等到事务结束）。X锁仍持续到事务结束。
   ​​*作用*​​：解决​​丢失更新​​和​​污读​​（禁止事务读取未提交的数据）。
   ​​*缺陷*​​：不解决不可重复读（S锁过早释放）
3. **三级封锁协议（解决不可重复读）​​**
   ​​*规则*​​：在一级基础上，​​读数据前加S锁，且S锁持续到事务结束​​（提交或回滚时释放）。X锁同样持续到事务结束。
   ​​*作用*​​：解决​​丢失更新​​、​​脏读​​、​​不可重复读​​（同一事务内多次读取相同数据时，S锁防止其他事务修改数据）。
   ​​*缺陷*​​：可能导致锁竞争增加，降低并发性能。

##### 封锁粒度

- 封锁粒度越小，系统中能被封锁的对象越多，并发度越高，但封锁机构越复杂，系统开销越大。
- 封锁粒度越大，系统中能被封锁的对象越少，并发度越低，但封锁机构越简单，系统开销越小。

##### 死锁和活锁

1. **活锁（Livelock）**
   ​*定义*​​：事务​​不断重试操作​​，但​​因外部干扰永远无法成功​​，处于“忙等”状态。
   *形象比喻*​​：两人同时侧身让路，结果反复同步动作，始终无法通过走廊。
2. **死锁（Deadlock）​**
   *定义*​​：多个事务​​互相持有对方需要的资源​​，且​​相互等待对方释放资源​​，形成永久阻塞状态。
   *形象比喻*​​：两人在狭窄走廊迎面相遇，都要求对方让路，结果谁都无法通过。

### 第七章 SQL Server高级应用

#### 变量及其使用（P272）

1. 全局变量：
   - 使用 `@@` 前缀开头
   - 全局变量不是用户自己定义的，而是SQL Server事先定义的。
   - 用户只能使用SQL Server事先定义的全局变量。因此全局变量对用户而言是只读的。
   - 全局变量由SQL Server自动维护。

2. 局部变量：
   - 使用 `@` 前缀开头
   - 局部变量必须先用`DECLARE`语句声明后才可以使用。其形式如下：
     DECLARE @变量1 变量类型及长度 \[=初值];
   - 赋值：
     - SET（单变量赋值）
       SET @变量名 = 值;
     - SELECT（查询结果赋值，可多变量）
       SELECT @变量名1 = 列1, @变量名2 = 列2 FROM ... WHERE ...;

   - 示例

```sql
DECLARE @MinPrice MONEY, @MaxPrice MONEY;

-- 多变量赋值（推荐 SELECT）
SELECT
    @MinPrice = MIN(Price),
    @MaxPrice = MAX(Price)
FROM Products;

-- 打印结果
PRINT '最低价格: ' + CAST(@MinPrice AS VARCHAR);
PRINT '最高价格: ' + CAST(@MaxPrice AS VARCHAR);

-- 条件查询
IF @MaxPrice > 1000
    PRINT '存在高价商品！';
```
